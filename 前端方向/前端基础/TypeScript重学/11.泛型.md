## 泛型定义函数的两种区别

泛型接口：

```ts
interface GenericIdentityFn<T> {
  (arg: T): T;
}

// 泛型参数属于接口，一单指定了泛型参数，那么接口中的所有参数都会使用这个泛型参数，函数参数在调用时就已经确定不会再进行类型推导
function identity<T>(arg: T): T {
  return arg;
}

let myIdentity: GenericIdentityFn<number> = identity;
myIdentity(0); // ok
myIdentity("0"); // error：类型不兼容，无法将字符串赋给数字
```

泛型函数：

```ts
interface GenericIdentityFn {
  <T>(arg: T): T;
}

// 泛型参数属于函数，在进行函数调用时会进行类型推导
function identity<T>(arg: T): T {
  return arg;
}

let myIdentity: GenericIdentityFn = identity;
myIdentity(0); // ok: T = number
myIdentity("0"); // ok: T = string
myIdentity<number>(0); // ok : 仍然可以显式指定泛型参数，注意，这时的泛型参数作用于函数上，而不是Interface上
```

### 混合使用

```ts
interface GenericIdentityFn<K> {
  <T>(arg: T): T;
  desc: K;
}

// 泛型参数属于函数，在进行函数调用时会进行类型推导
function identity<T>(arg: T): T {
  return arg;
}

identity.desc = "aha ~";

// 此处泛型应用于接口上
let myIdentity: GenericIdentityFn<string> = identity;
myIdentity<number>(0); // ok : 仍然可以显式指定泛型参数，注意，这时的泛型参数作用于函数上，而不是Interface上
console.log(myIdentity.desc);
```

## 泛型类

```ts
class GenericNumber<NumType> {
  zeroValue!: NumType;
  add!: (x: NumType, y: NumType) => NumType;
}

// 通过泛型参数指定类型，约束泛型类内部的泛型参数的类型
let myGenericNumber = new GenericNumber<number>();
myGenericNumber.zeroValue = 0;
myGenericNumber.add = function (x, y) {
  return x + y;
};
```

## 在泛型参数中使用类型参数约束

```ts
// 约束Key的类型必须为Type的属性
function getProperty<Type, Key extends keyof Type>(obj: Type, key: Key) {
  return obj[key];
}

let x = { a: 1, b: 2, c: 3, d: 4 };

getProperty(x, "a");
getProperty(x, "m"); // Argument of type '"m"' is not assignable to parameter of type '"a" | "b" | "c" | "d"'.
```

## 在泛型中使用类类型

```ts
class BeeKeeper {
  hasMask: boolean = true;

// 接收类作为参数
function create<Type>(c: { new (): Type }): Type {
  return new c();
}

createInstance(BeeKeeper).hasMask;
```

## 默认泛型参数

使用默认泛型参数可以简化一些泛型代码：

```ts
// 不带任何参数，生成DivElement的Container
// 带一个参数，生成指定元素类型的Container
// 带两个参数，生成指定元素类型容器和指定子元素数组类型的Container
declare function create(): Container<HTMLDivElement, HTMLDivElement[]>;
declare function create<T extends HTMLElement>(element: T): Container<T, T[]>;
declare function create<T extends HTMLElement, U extends HTMLElement>(
  element: T,
  children: U[],
): Container<T, U[]>;

// 使用默认泛型参数简化，将三种调用形式变为一种
declare function create<
  T extends HTMLElement = HTMLDivElement,
  U extends HTMLElement[] = T[],
>(element?: T, children?: U): Container<T, U>;
```

## 泛型类型多态模式下参数的协变、逆变、双变问题

协变（类型扩大）：

```ts
interface Box<T> {
  content: T;
}
interface Animal {}
interface Dog extends Animal {}

let boxA: Box<Dog> = { content: {} };
// 此处boxA 可以赋值给 boxB，因为Dog 是 Animal 的子类，因此推断出Box<Dog> 可以赋值给 Box<Animal>，这叫协变
let boxB: Box<Animal> = boxA;
```

逆变（类型收窄）：

```ts
interface Animal {
  name: string;
}
interface Dog extends Animal {
  legs: number;
}

interface Handler<T> {
  handle: (value: T) => void;
}

// 声明一个处理器，要求接收Dog类型的参数
const ha: Handler<Dog> = {
  handle: (val) => {
    console.log(val);
  },
};

// ERROR：尝试将Dog类型的处理器赋值给Animal类型的处理器，会报错，因为Animal不是Dog的子类，因此无法赋值
const hb: Handler<Animal> = ha;
// 处理器实际执行时要求传入Animal类型，如果上面一行是合法的，则下面这一行就存在危险
// 参数类型逆变就是用来处理这个情况的
hb.handle({
  name: "QQ",
});

// 反之，像下面这种逆变就是Ok的
// 处理器要求传入Animal类型
const hb: Handler<Animal> = {
  handle: (val) => {
    console.log(val);
  },
};
// 逆变：Animal => Dog，允许赋值，类型向下，收窄
const ha: Handler<Dog> = hb;
// 实际调用：传入的是更窄的类型，没有风险
ha.handle({
  name: "QQ",
  legs: 4,
});
```

双变：

```ts
interface Handler<T> {
  // 原先正确出发协变逆变验证的写法
  // handle: (value: T) => void;
  // 这会引发TS使用双变进行检查，导致验证失效
  handle(value: T): void;
}

const ha: Handler<Dog> = {
  handle: (val) => {
    console.log(val);
  },
};

// 此处应该报错，因为Animal不是Dog的子类，但是由于我们使用了函数写法，TS采用双变策略进行验证，因此不会报错
const hb: Handler<Animal> = ha;

// 此处有问题，Handler实际要求传入Dog类型，但是被赋值给Animal类型，并且TS类型检查由于采用双变策略，因此不会报错
// 这会儿产生一个运行时错误，编译时无法检查到
hb.handle({
  name: "QQ",
});
```

### 场景总结

| 场景                   | TS行为   | 原因                   |
| ---------------------- | -------- | ---------------------- |
| `readonly T[]`         | 协变     | 只读                   |
| `Array<T>`             | 协变     | 历史兼容（不完全安全） |
| 函数返回值             | 协变     | 输出安全               |
| 函数参数               | 逆变     | 输入安全               |
| 函数属性 `(x:T)=>void` | 逆变     | strictFunctionTypes    |
| 方法 `(x:T)=>void`     | 双变     | 兼容 JS                |
| 读写属性               | 通常不变 | 输入输出冲突           |

一个很典型的TS协变设计妥协：

```ts
interface Animal {
  name: string;
}

interface Dog extends Animal {
  legs: number;
}

const dogs: Dog[] = [{ name: "dog", legs: 4 }];
// 协变：Array<Dog> -> Array<Animal>，两个变量共享同一片内存地址，这埋下隐患
const animals: Animal[] = dogs;
// 使用协变类型放入宽类型，这会改变窄类型数组，造成错误
animals.push({
  name: "cat",
});
// 已经被前一步污染，无法取到有效值
console.log(dogs[1].legs);
```

## 泛型变动注解 Variance Annotations

泛型类型的变动类型是协变还是逆变还是不变，一般交给编译器处理就行了，绝大部分情况下不需要手动标注，但是下面两种场景可以使用手动标注来提速类型验证或者提升类型可读性与保护设计模式：

1. 递归类型：可能会导致TS类型系统递归检查造成缓慢，可以标注泛型为协变（out），进而加快检查速度，TS不再需要将类型进行展开进行分析了
2. 库API类型设计：可以为泛型类型标注为协变（in）或者逆变（out），从而提升类型可读性，保护设计模式（防止后来的维护者无意间破坏泛型的用途）

```ts
interface Handler<in T, out R> {
  // 设计者代码
  handler: (val: T) => void;
  get: () => R;

  // 修改者代码：
  run: (opt: R) => void;
  // 加入这一修改后，泛型 out R 会报错，因为 R 是逆变，但是TS检查发现 R 应该是 in + out，所以无法通过验证
  // 小声BB：报错信息没有任何可读性，对于不知道 Variance Annotations 的人来说：这TM什么玩意儿？？？
}
```
