## 基础案例

```ts
class Person {
  name = "name";
  // 如果开TS启了：--strictPropertyInitialization 标志位，则必须初始化或者在构造函数中完成初始化，否则会报错，不能使用后初始化的模式
  // 如果需要使用后初始化，则必须关闭这个标准位或者使用断言运算符来告知TS不要校验
  age: number;

  constructor(age: number) {
    this.age = age;
  }

  toString() {
    return `name: ${this.name}, age: ${this.age}`;
  }
}

const alice = new Person(22);
alice.name = "alice";

console.log(`${alice}`);
```

_Tips: TypeScript 不会分析从构造函数调用的方法来检测初始化情况，因为派生类可能会重写这些方法，导致成员初始化失败。_

如果确定需要使用构造函数意外的初始化方式来完成初始化：

```ts
class Person {
  name!: string;
}

const alice = new Person();
alice.name = "alice";
```

## 只读属性

使用`readonly`关键字来定义只读属性，防止构造函数之外的上下文修改对象属性，另外，TS会检查只读属性是否正确完成了初始化，如果没有初始化，则会报错，方法包括：属性直接初始化以及构造函数初始化

```ts
class Greeter {
  // 属性直接初始化
  readonly name: string = "world";
  // 构造函数条件初始化，需要注意的是，属性的只读状态对构造函数并不起作用
  // 如果属性没有直接初始化，则构造函数必须完成初始化，否则会报错，TS会分析构造函数是否在所有情况下都完成了初始化，如果存在未初始化的可能，则会报错
  constructor(otherName?: string) {
    if (otherName !== undefined) {
      this.name = otherName;
    }
  }
}
```

像下面这种，TS会告知你，属性没有正确完成初始化：

```ts
class Greeter {
  // 没有直接初始化
  readonly name: string;

  // 构造函数中，条件初始化，因此RS认定属性没有正确完成初始化
  constructor(otherName?: string) {
    if (otherName !== undefined) {
      this.name = otherName;
    }
  }
}
```

## extends 与 Super Call（继承与父类初始化）

子类型在构造函数中调用`super()`之前，不能访问`this`。

```ts
class Base {
  a = 4;
}

class Child extends Base {
  b: string;

  constructor() {
    // ERROR：'super' must be called before accessing 'this' in the constructor of a derived class.(17009)
    this.b = "aha~";
    super();
  }
}
```

## Getter & Setter

```ts
class Calctor {
  value: number;

  constructor(v: number) {
    this.value = v;
  }

  // 使用Setter计算圆面积
  get circularArea(): number {
    return Math.PI * this.value ** 2;
  }

  // 使用Setter检查输入值
  set checkValue(val: number) {
    if (val < 0) {
      throw new Error("This value cannot be less than 0.");
    }
    this.value = val;
  }
}

const cc = new Calctor(1);

cc.checkValue = 2;
console.log("PI*R^2: %s", cc.circularArea);
// 异常，无法通过Setter的检查
cc.checkValue = -1;
```

特殊规则：

1. 如果 get 属性存在但 set 不存在，则该属性自动 readonly
2. 如果未指定 setter 参数的类型，则会根据 getter 的返回类型推断
3. 自 TypeScript 4.3 起，可以使用不同类型的访问器进行获取和设置

## 索引签名

Class也支持索引属性签名，但是需要同时处理Class方法，很难有效利用这些类型，所以最佳实践是将索引数据类型放到其它地方，而不是Class本身。

```ts
class MyClass {
  [s: string]: boolean | ((s: string) => boolean);

  check(s: string) {
    return this[s] as boolean;
  }
}
```

## 接口实现

Class 可以实现接口，如果接口没有实现接口中的所有方法，则会引发TS报错：

```ts
interface Checkable {
  name: string;
  opt?: string;
  check(name: string): boolean;
}

class NameChecker implements Checkable {
  // 必须定义，因为 Checkable 中对这个属性进行了要求
  name: string = "name";
  // opt?: string; 可以不定义，因为 Checkable 中这个属性是可选的

  // 错误：类型不兼容，Class实现的类型不满足Interface约束
  // check(name: number): boolean { return true; }

  // 正确：类型兼容
  check(name: string): boolean {
    return true;
  }
}
```

**重点：class中使用 implements 实现接口，接口的类型并不会影响Class的类型定义，TS只是判断Class的实现是否满足 Interface 的约束，如果不满足就报错，仅此而已。**

## 属性与方法覆盖

```ts
class Base {
  greet() {
    console.log("Hello, world!");
  }
}

class Derived extends Base {
  greet(name?: string) {
    if (name) {
      console.log(`Hello, ${name}.`);
      return;
    }

    super.greet();
  }
}

const d = new Derived();

d.greet(); // 父类方法
d.greet("Alice"); // 子类方法

// 此处表现出对象的多态性质，子类实例可以赋值给父类实例，并且丢失子类型的私有实现签名，因此不能在调用带有参数数的greet
const b: Base = d;
b.greet("aaa"); // ERROR：Expected 0 arguments, but got 1.  因为带参实现是子类自己Override的，父类没有这个实现，就算这个实例是子类也不行，类型签名不匹配
```

## Type-only Field Declarations (类型字段声明)

类型字段声明用于子类型中的类型收窄，在子类型中，明确知道相关的属性是子类型，也希望使用父类型的初始化流程，体现继承关系时，就需要用到这个特性：

```ts
interface Animal {
  dateOfBirth: any;
}

interface Dog extends Animal {
  breed: any;
}

class AnimalHouse {
  resident: Animal;
  constructor(animal: Animal) {
    this.resident = animal;
  }
}

class DogHouse extends AnimalHouse {
  // 此处使用 declare 对 resident 进行类型收窄，然后在子类型的其他方法中就可以正确知晓 resident 是 Dog 类型，而不是更宽泛的 Animal 类型
  // declare 只存在于类型系统，编译后完全消失，如果不使用，则相当于复写了父类型的同名属性，TS会要求完成初始化，这可能会添加重复初始化逻辑或破坏继承结构
  declare resident: Dog;
  constructor(dog: Dog) {
    super(dog);
  }
}
```

如果不使用 declare，直接覆盖声明，就会出现两种情况：

1. 重新在构造函数中初始化
2. 直接不初始化

上面两种处理方式均有问题：

```ts
class DogHouse extends AnimalHouse {
  resident: Dog;
  constructor(dog: Dog) {
    super(dog);
    // 第一种
    // 重复初始化，这个逻辑已经在父类中完成，这里为了通过TS验证，只能添加重复逻辑
    // 另外，如果父类型初始化逻辑并非这么简单的话，可能会破坏继承结构
    this.resident = dog;

    // 第二种
    // 直接不初始化，一般来说TS会报错，告知你没有完成初始化，但是如果你关闭了验证，则生成的JS代码中，resident 属性会先被父类型完成初始化
    // 然后再被子类型重新覆盖为 undefined，最终导致，这个属性始终是 undefined，进而造成BUG
  }
}
```

declare 属性声明就是干这个用的，仅用作类型收窄，不影响实际运行逻辑。

## 字段初始化与构造函数运行时机

```ts
class Base {
  name = "base";
  constructor() {
    console.log("My name is " + this.name);
  }
}

class Derived extends Base {
  name = "derived";
}

// Prints "base", not "derived"
const d = new Derived();
```

运行实际顺序如下：

1. 基类字段初始化
2. 基类构造函数执行
3. 子类字段初始化
4. 子类构造函数执行

因此最终结果将是：My name is base，这也解释了上一个例子中如果不使用 declare 为什么最终属性将会是 undefined

## 成员可见性

类成员的默认可见性为 public，public 成员可以在任何地方访问，因为 public 已经是默认的可见性修饰符，所以你永远不需要在类成员上写它，但出于风格/可读性的考虑，你可能会选择这样做。

- public：允许从类外部访问成员
- protected：允许从类内部或继承的子类访问，禁止从外部访问
- private：仅允许从类内部访问

## 使用 使用Public 公开 Protected成员

```ts
class Base {
  protected m = 10;
}
class Derived extends Base {
  // 使用Override复写父类成员来公开父类成员
  public m: number = 15;
  // 更推荐的写法，如果只是希望变更成员可见性，应该是用 declare 来声明成员可见性，而不是 Override
  declare public m: number;
}
const d = new Derived();
console.log(d.m); // OK
```

下面这些写法通通不允许：

```ts
class Base {
  protected m = 10;
}
class Derived extends Base {
  // 尝试收窄类型，不允许
  declare private m: number;
}

// 详解：由于多态的存在，子类实例可以赋值给父类实例，然后当父类访问属性时，属性被子类收窄为Private，此时父类就无法访问了，逻辑冲突，因此不允许这种类型收窄

class Base {
  private m = 10;
}
class Derived extends Base {
  // 尝试放宽一个在子类型不可见的类型，不允许
  // Property 'm' is private in type 'Base' but not in type 'Derived'.
  declare protected m: number;

  // 尝试使用覆盖父类成员来公开父类成员，不允许
  // Class类型错误：Property 'm' is private in type 'Base' but not in type 'Derived'.
  public m: number = 15;
}

// 详解：declare protected m: number;
// 乍一看好像没什么问题，实际上不可以，我们需要从类的角度来看，Base中定义的m属性为private类型，在Derived中不可见，TS会告诉你
// Derived 中压根儿没有这个类型，因此想要调整属性声明就更不可能了。

// 详解：public m: number = 15;
// 这个解释起来就更麻烦了，逻辑上其实这样写是可以的，并且在JS中确实是可行的，JS会进行变量遮蔽（覆盖），子类的值会覆盖父类的值，但是在TS中行不通
// TS 不允许这种变量遮蔽，这会造成语义混乱，看下面：

class Base {
  private m = 10;

  print() {
    // 这里到底输出 10 还是 20？
    console.log(this.m);
  }
}
class Derived extends Base {
  protected m = 20;
}

// 在JS中，实际会输出20，子类进行了值覆盖，但是从类型系统来看，一个private属性，只能被类内部访问，结果实际上这个值被外部修改了，这是不允许的，
// 这就是TS的设计取舍（限制一些合理语法换取设计上的清晰）
```

由于 private 成员对派生类不可见，因此派生类无法提高其可见性。

## 误区：protected 类型间属性访问

```ts
class Base {
  protected x: number = 1;
}
class Derived1 extends Base {
  protected x: number = 5;
}
class Derived2 extends Base {
  f1(other: Derived2) {
    other.x = 10;
  }
  f2(other: Derived1) {
    // Property 'x' is protected and only accessible within class 'Derived1' and its subclasses.
    other.x = 10;
  }

  f3(other: Base) {
    // Property 'x' is protected and only accessible within class 'Derived1' and its subclasses.
    other.x = 10;
  }
}
```

受保护的属性只能被类本身和子类访问，Derived2 并不是 Derived1 的子类，因此不允许访问，f3 中同理，Base也不是 Derived1 的子类，因此也不允许访问。

## 跨实例 private 访问

private 限定的是类，不是类实例，主流OOP语言都是这种模式，Ruby是特例。

```ts
class A {
  private x = 10;
  // 如果不允许，那下面这种正常的功能就无法实现，会报错：不能从 other 访问一个私有属性 x
  public sameAs(other: A) {
    // No error

  }

```

## TS Private 为软私有

TS 中 private 还允许在类型检查期间使用方括号表示法访问。这使得声明为 private 字段在单元测试等情况下更容易访问，但缺点是这些字段是软私有的 ，并不严格强制执行隐私性。这与JS中的私有成员有本质区别，JS中的私有成员是真私有，任何外部手段都无法访问到。

```ts
class A {
  private x = 10;
}
const a = new A();
console.log(a["x"]); // ok，啊，这...
```

使用 ES2022 的私有属性语法，创建硬私有成员：

```ts
class A {
  // 私有属性语法不能与可见性修饰符一起使用
  #x = 10;

  getX() {
    return this.#x;
  }
}
const a = new A();
console.log(a.getX()); // ok
console.log(a); // 无法观察到 #x 属性
console.log(a["#x"]); // ERROR：Property '#x' does not exist on type 'A'.
```

## 成员函数this指向问题处理

普通函数在上下文变更时，this 指向会发生改变：

```ts
class A {
  name = "A";

  getName() {
    return this.name;
  }
}

const a1 = new A();
const obj = {
  name: "obj",
  // getName this 指向发生改变
  getName: a1.getName,
};
// obj
console.log(obj.getName());
```

### 解决方式1：使用箭头函数，创建闭包，固定 this 指向

```ts
class A {
  name = "A";
  getName = () => {
    return this.name;
  };
}
```

会生成如下JS代码，可以观察到每次创建实例时，都会为getName属性创建一个新函数，该函数捕获了外层this，因此在调用时，this 始终指向实例：

```js
"use strict";
class A {
  constructor() {
    this.name = "A";
    this.getName = () => {
      return this.name;
    };
  }
}
```

_Tips：如果一个类需要创建大量实例，不建议用这种方法，每次创建实例都会创建一个函数闭包，这会导致大量的内存占用_

### 解决方式2：使用TS类型系统进行约束，为成员方法指定this类型

```ts
class A {
  name = "A";
  getName(this: A) {
    return this.name;
  }
}

const a1 = new A();
const fn = a1.getName;

// error: The 'this' context of type 'void' is not assignable to method's 'this' of type 'A'.
console.log(fn());
```

但是需要注意的是，TS类型系统只是检查调用对象是否符合类型约束，而不是检查一个对象真的是某个类的实例（obj instanceof Class），只要符合类型约束，就可以调用该方法:

```ts
const a1 = new A();
// 与Class A 完全兼容，TS 认为obj 就是Class A，因此在obj上调用getName方法不会报错
const obj = {
  name: "obj",
  getName: a1.getName,
};

console.log(obj.getName());
```

如果我们不想要这种行为，要求方法只能在类的实例上调用，可以借助私有成员来完成控制：

```ts
class A {
  // 添加一个空私有成员，用来限制调用者只能在类的实例上调用，因为外部无法创建这个私有成员
  private __brand!: void;
  name = "A";
  getName(this: A) {
    return this.name;
  }
}

// other code...

// ERROR：The 'this' context of type '{ name: string; getName: (this: A) => string; }' is not assignable to method's 'this' of type 'A'.
// Property '__brand' is missing in type '{ name: string; getName: (this: A) => string; }' but required in type 'A'.
console.log(obj.getName());
```

## This 类型

在类中，有一种名为 this 特殊类型，它会动态地引用当前类的类型。

```ts
class Box {
  contents: string = "";
  set(value: string) {
    this.contents = value;
    // 返回this实现链式调用
    return this;
  }

  // 此处This将始终推断为实际的Class类型，而不是固定为Box类型，因此，如果Box具备子类，则这个This在子类调用中会被推断为子类类型
  sameAs(other: this) {
    return other.contents === this.contents;
  }
}

const base = new Box();
const derived = new DerivedBox();
derived.sameAs(base); // Error：父类无法赋值给子类
base.sameAs(derived); // Ok：子类可以赋值给父类

// 如果希望sameAs可以基于定义的类进行工作，可以将This类型更换为固定类的父类型即可：
class Box {
  contents: string = "";
  sameAs(other: Box) {
    // 参数类型始终为Box，类型继承后也不会改变
    return other.contents === this.contents;
  }
}
```

## 构造函数参数属性

TypeScript 提供了一种特殊的语法，可以将构造函数参数转换为同名同值的类属性。这些属性称为参数属性 ，创建方法是在构造函数参数前加上 public 、 private 、 protected 或 readonly 等可见性修饰符。生成的字段会带有这些修饰符。

这个方式可以简化Class定义语法，但是会降低可读性。

```ts
class Params {
  constructor(
    public readonly x: number,
    protected y: number,
    private z: number,
  ) {
    // 不再需要在构造函数中赋值
  }
}
const a = new Params(1, 2, 3);
console.log(a.x);
```

```ts
const someClass = class<Type> {

const m = new someClass("Hello, world");
```

## `InstanceType<T>` 类型

`InstanceType<T>` 类型用于获取构造函数 T 的实例类型。

这是一个非常绕的概念，首先，TS将符号表分成了两个空间：值空间、类型空间，然后 `const objA = create(A, 10);` 此处的 A 位于表达式中，因此查询值空间的A，也就是构造函数 A。

检查是否满足泛型约束，显然，A是构造函数并满足约束，此时 T 被推断为：`typeof A` 也就是 `new (...args: number[]) => A`。

`InstanceType<T>` 展开写为：`InstanceType<T> = T extends new (...args: any[]) => infer R ? R : never`

工厂函数拿到了构造函数的签名，但是不知道构造函数返回的实例是什么类型，因此需要从构造函数反推出实例类型，`InstanceType<T>` 就是干这个用的，根据`InstanceType<T>`的定义展开，可以观察到，内部使用了`infer R` 来获取构造函数的返回类型。

另外，`ConstructorParameters<T>` 类型用于获取构造函数 T 的参数列表类型，这两个工具类型通常成对出现。

```ts
class A {
  constructor(public x: number) {}
}

// 使用构造函数签名，约束T为类构造函数类型
function create<T extends new (...args: any[]) => any>(
  ctor: T,
  // 提取构造函数的参数表类型，用做工厂函数的参数类型定义
  ...args: ConstructorParameters<T>
  // 从构造函数反推出实例类型
): InstanceType<T> {
  return new ctor(...args);
}

const objA = create(A, 10);

console.log(objA.x);
```

更加严谨的写法，构造函数中的类型将会正确推断，区别不大：

```ts
type CtorFn<T extends new (...args: any[]) => any> = new (
  ...args: ConstructorParameters<T>
) => InstanceType<T>;

function create<T extends CtorFn<T>>(
  ctor: T,
  ...args: ConstructorParameters<T>
): InstanceType<T> {
  return new ctor(...args);
}
```
