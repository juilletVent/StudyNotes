## 命名空间基础背景

命名空间是TS自己的概念，用于JS的模块化系统未完善的时代，现在JS的模块系统已经非常完善了，不再需要TS的命名空间来组织模块化了，但是namespace依然有一定的价值：

1. 全局类型声明的隔离
2. 为全局对象进行扩展

## 为项目内js文件补充类型定义

```js
// add.js
export function add(a, b) {
  return a + b;
}
```

```ts
// 为add增加类型定义
// add.d.ts
export function add(a: number, b: number): number;
```

使用时，正常引入js即可：

```ts
import { add } from "./add.js";
add(1, 2);
```

## 为项目内的第三方库补充类型定义

1. 在项目中创建 types/ 目录
2. 在里面新建 xxx.d.ts（文件名随意）
3. 用 declare module '包名' { ... } 声明你用到的 API
4. 确保 tsconfig.json 能扫到这个目录：
   - 方式A：types/ 在 tsconfig "include" 范围内
   - 方式B：配置 "typeRoots": ["./node_modules/@types", "./types"]
5. 直接在代码中 import ... from '包名'，TS 就有提示了

⚠️ 注意：

- declare module 中的包名必须和 npm 包名 / import 路径完全一致
- 不用声明全部 API，只声明你用到的就行（TS 不会检查运行时是否存在）
- 如果社区后来发布了 @types/包名，记得卸载并删掉自己的声明

```ts
// 假设库名称：legacy-parser
declare module "legacy-parser" {
  interface ParseOptions {
    strict?: boolean;
    maxDepth?: number;
  }

  interface ParseResult {
    ast: unknown;
    warnings: string[];
  }

  /** 默认导出：主解析函数 */
  export function parse(input: string, options?: ParseOptions): ParseResult;

  /** 命名导出：版本号 */
  export const version: string;
}
```

## 为全局对象定义扩展属性或方法

如果需要为Window对象或者global添加额外的全局属性或方法，可以使用下面方法：

```ts
// 方法一：借助Interface合并，将自定义的Window接口与预设的Window接口进行合并即可完成
// global.d.ts 文件内不能有 import export
interface Window {
  sayHello: () => string;
}

// 方法二：如果文件是模块文件，则需要定义全局模块来完成
// xxx.d.ts
// 通过定义全局模块，在模块内仍然能够完成全局对象的扩展
declare global {
  interface Window {
    sayHello: () => string;
  }
}

function add(a: number, b: number) {
  return a + b;
}

export { add };
```

## 全局裸变量

```js
var global = "global";
```

对应TS声明

```ts
declare var global: string;
```

## 为资源模块声明类型

告知TS所有以jpg结尾的模块，默认导出都是 string，实际处理工作交给bundler（webpack、vite、rspack等）完成：

```ts
declare module "*.jpg" {
  const src: string;
  export default src;
}
```
